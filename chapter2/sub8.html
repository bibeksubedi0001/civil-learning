Chapter 2 — Sub-chapter 2.8

Model Evaluation Metrics

A model that predicts every bridge is safe is 99% accurate — until the one failure it missed causes a catastrophe. Evaluation metrics tell you not just how often your model is right, but in what ways it fails. For civil engineers, choosing the right metric is as critical as choosing the right steel grade: it determines whether your AI system prioritises safety, cost, or balanced performance.

45 min read 2 interactive demos 4 quiz questions

The Accuracy Trap

Imagine an AI model deployed to scan thousands of concrete bridge deck images for cracks. Out of 10,000 images, only 50 contain actual cracks. A model that simply predicts "no crack" for every single image achieves 99.5% accuracy — a number that looks spectacular on a dashboard but is catastrophically useless in practice. It misses every single defect.

This is the accuracy paradox, and it is the reason civil engineers cannot rely on a single number to evaluate AI models. We need a toolkit of metrics, each illuminating a different aspect of model performance, and we need the engineering judgement to choose which metrics matter most for each specific application.

In this sub-chapter, we will build that toolkit from the ground up. We will cover classification metrics (accuracy, precision, recall, F1 score, ROC-AUC, and the confusion matrix), regression metrics (MAE, MSE, RMSE, R-squared, and adjusted R-squared), and the cross-validation techniques that give us reliable estimates of how a model will perform on data it has never seen. Every concept will be grounded in civil engineering examples.

In this sub-chapter, we will cover:

  • Why accuracy alone is misleading for imbalanced engineering datasets
  • The confusion matrix — the foundation of all classification metrics
  • Precision, recall, and the F1 score — understanding the safety-performance trade-off
  • ROC curves and AUC — threshold-independent model comparison
  • Regression metrics: MAE, MSE, RMSE, R-squared, and adjusted R-squared
  • Cross-validation: k-fold, stratified k-fold, and why a single train-test split is not enough
  • Choosing the right metric for safety-critical civil engineering applications

The Confusion Matrix: Foundation of Classification Metrics

Before we can define any classification metric, we need a systematic way to categorise predictions. The confusion matrix does exactly this. For a binary classifier (two classes), it is a 2x2 table that records four counts:

Binary Confusion Matrix 
                        Predicted: Positive    Predicted: Negative
                       +---------------------+---------------------+
  Actual: Positive     |  True Positive (TP)  | False Negative (FN) |
                       +---------------------+---------------------+
  Actual: Negative     | False Positive (FP)  | True Negative (TN)  |
                       +---------------------+---------------------+

Civil Engineering Example (Crack Detection):
  Positive = "Crack present"    Negative = "No crack"

  TP = Model says crack, and there IS a crack       (correct alert)
  FP = Model says crack, but there is NO crack      (false alarm)
  FN = Model says no crack, but there IS a crack    (MISSED DEFECT!)
  TN = Model says no crack, and there is NO crack   (correct clearance)

Notice the asymmetry in consequences. In structural safety, a false negative (missed crack) is far more dangerous than a false positive (unnecessary inspection). This asymmetry is the reason we need metrics beyond accuracy.

Civil Engineering Analogy

Think of the confusion matrix like a quality control report for weld inspections. The inspector examines 200 welds and classifies each as "defective" or "acceptable". The confusion matrix summarises: how many truly defective welds were correctly flagged, how many acceptable welds were incorrectly rejected, how many defective welds were missed, and how many acceptable welds were correctly approved. Each number tells a different story about the inspector's performance.

The total number of samples is simply TP + FP + FN + TN. From these four counts, we derive every classification metric that follows.

Accuracy: The Deceptive Starting Point

Accuracy Formula 
Accuracy = (TP + TN) / (TP + TN + FP + FN)

         = Number of correct predictions / Total predictions

Example: Crack detection on 10,000 images
  - 50 images have cracks, 9,950 do not
  - Model predicts "no crack" for ALL images:

  TP = 0, FP = 0, FN = 50, TN = 9,950

  Accuracy = (0 + 9,950) / (0 + 9,950 + 0 + 50)
           = 9,950 / 10,000
           = 99.5%        <-- Looks great, but misses EVERY crack!

Accuracy answers: "what fraction of all predictions were correct?" It treats all errors equally — a missed crack and a false alarm carry the same weight. This makes accuracy a reasonable metric only when:

  • The classes are roughly balanced (similar number of positives and negatives)
  • The costs of different errors are equal (false positive is as bad as false negative)

In civil engineering, neither condition is typically met. Structural failures are rare (imbalanced classes), and missing a failure is far costlier than a false alarm (asymmetric error costs). We need better metrics.

Precision: When You Predict Positive, How Often Are You Right?

Precision Formula 
Precision = TP / (TP + FP)

          = True Positives / All Positive Predictions

"Of all the welds I flagged as defective, how many truly were defective?"

Example: Model flags 30 welds as defective
  - 25 are truly defective (TP = 25)
  - 5 are actually fine    (FP = 5)

  Precision = 25 / (25 + 5) = 25/30 = 83.3%

Precision measures the trustworthiness of positive predictions. High precision means when the model raises an alarm, you can trust it. Low precision means the model cries wolf too often.

Precision matters most when false positives are expensive. For example, if every crack alert triggers a costly bridge closure and emergency inspection team deployment, you want your model to have high precision — you do not want to shut down a highway bridge for a shadow that looked like a crack.

Recall: How Many Actual Positives Did You Catch?

Recall Formula 
Recall = TP / (TP + FN)

       = True Positives / All Actual Positives

"Of all the welds that are truly defective, how many did I catch?"

Also called: Sensitivity, True Positive Rate (TPR), Hit Rate

Example: There are 40 truly defective welds
  - Model catches 35 of them  (TP = 35)
  - Model misses 5            (FN = 5)

  Recall = 35 / (35 + 5) = 35/40 = 87.5%

Recall measures completeness — what fraction of the actual positives did the model find? High recall means the model misses very few true positives. Low recall means many real defects slip through undetected.

Critical for Structural Safety

In safety-critical applications like crack detection, corrosion monitoring, or structural health assessment, recall is the most important metric. A missed defect (false negative) can lead to structural failure, injuries, or fatalities. It is far better to send an inspector to check a false alarm than to miss a real crack. In these contexts, aim for recall above 95% — even at the cost of lower precision.

Here is the fundamental tension: you can always achieve 100% recall by predicting "positive" for everything — but then precision drops to near zero. Conversely, you can achieve high precision by only predicting "positive" when you are absolutely certain — but then recall drops because you miss borderline cases. This is the precision-recall trade-off.

F1 Score: The Harmonic Mean Compromise

F1 Score Formula 
F1 = 2 * (Precision * Recall) / (Precision + Recall)

   = Harmonic mean of Precision and Recall

Why harmonic mean, not arithmetic mean?
  - Arithmetic mean of 100% and 0% = 50%  (misleadingly high)
  - Harmonic mean of 100% and 0% = 0%     (correctly penalises imbalance)

The harmonic mean punishes extreme imbalance between precision and recall.

Example:
  Precision = 83.3%,  Recall = 87.5%
  F1 = 2 * (0.833 * 0.875) / (0.833 + 0.875)
     = 2 * 0.729 / 1.708
     = 0.854 = 85.4%

The F1 score provides a single number that balances precision and recall. It is high only when both precision and recall are high. If either drops, F1 drops sharply.

However, the standard F1 score gives equal weight to precision and recall. In civil engineering, we often want to weight recall more heavily. The generalised F-beta score allows this:

F-beta Score 
F_beta = (1 + beta^2) * (Precision * Recall) / (beta^2 * Precision + Recall)

  beta = 1  -->  F1 (equal weight)
  beta = 2  -->  F2 (recall is twice as important as precision)
  beta = 0.5 -> F0.5 (precision is twice as important as recall)

For structural safety:  Use F2 (emphasise recall)
For cost optimisation:  Use F0.5 (emphasise precision)

ROC Curve and AUC: Threshold-Independent Evaluation

Most classifiers do not directly output a class label. Instead, they output a probability or score, and we apply a threshold to convert it into a prediction. For instance, a logistic regression model might output P(crack) = 0.72. If our threshold is 0.5, we predict "crack". If our threshold is 0.8, we predict "no crack".

The Receiver Operating Characteristic (ROC) curve plots the model's performance across all possible thresholds. It shows how the True Positive Rate (recall) changes relative to the False Positive Rate as we vary the threshold from 0 to 1.

ROC Curve Axes 
X-axis: False Positive Rate (FPR) = FP / (FP + TN)
        "Of all negatives, how many did we incorrectly flag?"

Y-axis: True Positive Rate (TPR) = TP / (TP + FN)  = Recall
        "Of all positives, how many did we correctly catch?"

Key reference lines:
  - Diagonal (TPR = FPR):  random guessing (coin flip)
  - Top-left corner (0,1): perfect classifier
  - Along the curve:       each point is a different threshold

AUC (Area Under the ROC Curve):
  - AUC = 1.0:   perfect model
  - AUC = 0.5:   random guessing
  - AUC < 0.5:   worse than random (predictions are inverted)

The beauty of the ROC curve is that it is threshold-independent. Instead of evaluating the model at a single threshold, it shows the full range of trade-offs available. The AUC summarises this into a single number: the probability that the model ranks a randomly chosen positive example higher than a randomly chosen negative example.

Civil Engineering Interpretation

Suppose you are comparing two models for classifying bridge structural condition ratings. Model A has AUC = 0.92 and Model B has AUC = 0.85. This means if you pick a random bridge that truly needs repair and a random bridge that is safe, Model A assigns a higher risk score to the unsafe bridge 92% of the time, while Model B does so only 85% of the time. Model A is better at ranking bridges by risk, regardless of what threshold you ultimately choose for triggering inspections.

One caveat: the ROC curve can be overly optimistic on highly imbalanced datasets. When negatives vastly outnumber positives (as in crack detection), even a large number of false positives barely moves the FPR. In such cases, the Precision-Recall curve (plotting precision vs. recall) gives a more informative picture.

Regression Metrics: Measuring Continuous Predictions

Classification metrics apply when we predict categories (crack / no crack, damage level 1-5). But many civil engineering models predict continuous values: concrete compressive strength in MPa, traffic flow in vehicles per hour, settlement in millimetres. For these, we need regression metrics that measure how far predictions deviate from actual values.

Mean Absolute Error (MAE)

MAE Formula 
MAE = (1/n) * SUM( |y_i - y_hat_i| )    for i = 1..n

Where:
  y_i     = actual value for sample i
  y_hat_i = predicted value for sample i
  n       = number of samples
  | . |   = absolute value

Example: Predicting concrete strength (MPa)
  Actual:    [30, 35, 28, 42, 38]
  Predicted: [32, 33, 30, 40, 36]
  Errors:    [ 2,  2,  2,  2,  2]

  MAE = (2 + 2 + 2 + 2 + 2) / 5 = 2.0 MPa

Interpretation: "On average, predictions are off by 2 MPa."

MAE is the simplest regression metric. It gives you the average magnitude of errors in the same units as the target variable. An MAE of 2 MPa for concrete strength is immediately interpretable by any structural engineer.

MAE treats all errors equally — an error of 10 MPa counts as five times worse than an error of 2 MPa, no more, no less. This makes MAE robust to outliers. A single wildly wrong prediction does not disproportionately inflate the metric.

Mean Squared Error (MSE) and Root Mean Squared Error (RMSE)

MSE and RMSE Formulas 
MSE  = (1/n) * SUM( (y_i - y_hat_i)^2 )

RMSE = sqrt(MSE)

Example: Predicting settlement (mm)
  Actual:    [12.0, 8.5, 15.2, 10.1, 22.0]
  Predicted: [11.5, 9.0, 14.0, 10.5, 18.0]
  Errors:    [ 0.5, 0.5,  1.2,  0.4,  4.0]
  Squared:   [0.25, 0.25, 1.44, 0.16, 16.0]

  MSE  = (0.25 + 0.25 + 1.44 + 0.16 + 16.0) / 5 = 3.62 mm^2
  RMSE = sqrt(3.62) = 1.90 mm

Compare to MAE = (0.5+0.5+1.2+0.4+4.0)/5 = 1.32 mm

Note: RMSE > MAE because squaring penalises the large error (4.0 mm) more heavily.

MSE squares each error before averaging, which means large errors are penalised disproportionately. An error of 4 mm contributes 16 to the sum, while an error of 1 mm contributes only 1. RMSE takes the square root of MSE to bring the result back to the original units (mm, MPa, etc.), making it more interpretable.

When should you prefer RMSE over MAE? Use RMSE when large errors are especially undesirable. In structural engineering, a settlement prediction that is off by 20 mm is not just four times worse than being off by 5 mm — it might mean the difference between acceptable performance and structural damage. RMSE captures this non-linear penalty.

R-Squared and Adjusted R-Squared

R-Squared Formula 
R^2 = 1 - (SS_res / SS_tot)

Where:
  SS_res = SUM( (y_i - y_hat_i)^2 )      (residual sum of squares)
  SS_tot = SUM( (y_i - y_mean)^2 )        (total sum of squares)
  y_mean = mean of actual values

Interpretation:
  R^2 = 1.0:  model explains ALL variance (perfect predictions)
  R^2 = 0.0:  model is no better than predicting the mean every time
  R^2 < 0.0:  model is WORSE than predicting the mean (possible!)

Example:
  Concrete strength model: R^2 = 0.92
  "The model explains 92% of the variance in concrete strength."

R-squared tells you what proportion of the variance in the target variable your model explains. Unlike MAE and RMSE, which depend on the scale of the data, R-squared is dimensionless and always interpretable: 0.92 means your model captures 92% of the variability in the data.

Adjusted R-Squared 
Adjusted R^2 = 1 - [(1 - R^2) * (n - 1) / (n - p - 1)]

Where:
  n = number of samples
  p = number of features (predictors)

Why adjusted?
  R^2 always increases (or stays the same) when you add more features,
  even if those features are irrelevant noise.
  Adjusted R^2 penalises for adding features that don't improve the model.

Example:
  Model with 5 features:     R^2 = 0.91,  Adj R^2 = 0.89
  Model with 20 features:    R^2 = 0.93,  Adj R^2 = 0.86
  --> The 20-feature model has higher R^2 but LOWER Adj R^2,
      suggesting many of those extra features are noise.

Adjusted R-squared is particularly important in civil engineering applications where you might be tempted to throw in every available measurement — cement content, water-cement ratio, aggregate size, curing temperature, humidity, admixture dosage, and so on. Adjusted R-squared tells you whether adding that 15th feature genuinely improves the model or just inflates the numbers.

Choosing the Right Metric for Civil Engineering

Selecting the right evaluation metric is an engineering decision, not a statistical one. It depends on the consequences of each type of error in your specific application. Here is a practical guide:

Metric Selection Guide 
APPLICATION                  | PRIMARY METRIC | REASON
-----------------------------+----------------+-------------------------------
Crack detection              | Recall (F2)    | Must not miss defects
Structural damage rating     | Recall + AUC   | Safety-critical, multi-class
Concrete strength prediction | RMSE + R^2     | Penalise large errors
Traffic flow forecasting     | MAE            | Errors are proportionally bad
Material classification      | F1 (macro)     | Balanced multi-class
Soil liquefaction screening   | Recall         | Conservative safety screening
Cost estimation              | MAE + MAPE     | Business metric, interpretable
Bridge load rating           | RMSE           | Large errors are dangerous
Earthquake damage prediction | AUC + Recall   | Imbalanced, safety-critical
The Golden Rule

In any safety-critical civil engineering application, default to recall (or equivalently, minimise false negatives). It is always cheaper to investigate a false alarm than to miss a real defect. Only optimise for precision or F1 when you have explicitly verified that false positives carry significant costs (e.g., unnecessary bridge closures costing millions per hour).

Cross-Validation: Reliable Performance Estimates

So far, we have discussed what to measure. Now we address how to measure it reliably. The naive approach — train on part of the data, test on the rest — gives you a single performance estimate that is highly dependent on which specific samples ended up in the training set versus the test set.

Imagine you have 100 concrete mix test results. You randomly put 80 in training and 20 in testing. If, by chance, the test set contains mostly easy-to-predict mixes, your RMSE looks great. If it contains the tricky outlier mixes, RMSE looks terrible. Neither estimate reflects the model's true capability.

Cross-validation solves this by systematically rotating which data serves as the test set, giving you multiple performance estimates that average out the luck of the draw.

K-Fold Cross-Validation

K-Fold Algorithm 
1. Shuffle the dataset randomly
2. Split it into K equal-sized "folds"
3. For each fold k = 1, 2, ..., K:
     a. Use fold k as the TEST set
     b. Use all other folds as the TRAINING set
     c. Train the model, evaluate on fold k, record the score
4. Final score = average of K scores
   Uncertainty  = standard deviation of K scores

Common choices:
  K = 5   (default for most applications)
  K = 10  (more reliable but slower)
  K = n   (Leave-One-Out; expensive, low bias, high variance)

Example with K=5 and 100 samples:
  Fold 1: Train on samples 21-100, test on 1-20   --> RMSE = 2.1
  Fold 2: Train on samples 1-20, 41-100, test 21-40 --> RMSE = 2.4
  Fold 3: ...                                      --> RMSE = 1.9
  Fold 4: ...                                      --> RMSE = 2.3
  Fold 5: ...                                      --> RMSE = 2.0

  Mean RMSE = 2.14 +/- 0.19

K-fold cross-validation gives you both a point estimate (the mean) and a confidence interval (via the standard deviation). If the scores vary wildly across folds, it signals that your model's performance depends heavily on which data it sees — a red flag for robustness.

Stratified K-Fold: Preserving Class Proportions

Standard k-fold randomly assigns samples to folds, which can cause problems with imbalanced datasets. If only 5% of your samples are "cracked", a random split might put all cracked samples into 2 of the 5 folds, leaving 3 folds with no positive examples at all — making those folds useless for evaluating recall.

Stratified vs Standard 
Dataset: 100 samples, 10 cracked (10%), 90 intact (90%)

Standard 5-fold (random):
  Fold 1: 3 cracked, 17 intact  (15% cracked)
  Fold 2: 0 cracked, 20 intact  ( 0% cracked)  <-- No positives!
  Fold 3: 4 cracked, 16 intact  (20% cracked)
  Fold 4: 1 cracked, 19 intact  ( 5% cracked)
  Fold 5: 2 cracked, 18 intact  (10% cracked)

Stratified 5-fold (preserves proportions):
  Fold 1: 2 cracked, 18 intact  (10% cracked)
  Fold 2: 2 cracked, 18 intact  (10% cracked)
  Fold 3: 2 cracked, 18 intact  (10% cracked)
  Fold 4: 2 cracked, 18 intact  (10% cracked)
  Fold 5: 2 cracked, 18 intact  (10% cracked)

Each fold mirrors the overall class distribution!

Stratified k-fold ensures each fold has approximately the same percentage of each class as the full dataset. This is essential for civil engineering datasets where the class of interest (defects, failures, anomalies) is typically rare. Always use stratified k-fold for classification problems with imbalanced classes.

Putting It All Together: A Complete Evaluation Workflow

Here is the recommended workflow for evaluating any civil engineering ML model:

Evaluation Workflow 
Step 1: Choose your primary metric based on error consequences
        - Safety-critical? --> Recall or F2
        - Continuous prediction? --> RMSE (if large errors are bad) or MAE

Step 2: Select secondary metrics for a fuller picture
        - Classification: always report confusion matrix + AUC
        - Regression: report MAE, RMSE, and R^2 together

Step 3: Use stratified k-fold cross-validation (k=5 or k=10)
        - Report mean +/- standard deviation
        - Check if variance across folds is acceptable

Step 4: Compare models using the SAME cross-validation splits
        - Use the same random seed for fair comparison

Step 5: Report results in engineering units
        - "RMSE = 1.9 mm" not just "RMSE = 1.9"
        - Include confidence intervals

Step 6: Validate against domain knowledge
        - Is an MAE of 3 MPa acceptable for your application?
        - Ask: would a structural engineer trust these predictions?

Reality Check

A model with R-squared = 0.95 sounds impressive, but if the RMSE is 8 MPa for a concrete strength prediction task where the acceptable tolerance is 3 MPa, the model is not fit for purpose. Always translate metrics back into engineering terms and compare against the tolerances specified in your design codes (e.g., ACI 318, Eurocode 2, IS 456).

CE Applications: Choosing Metrics That Matter

The "right" metric depends entirely on the engineering consequence of being wrong. Here's how to match metrics to real civil engineering problems.

Geotechnical: Metric Selection Guide

  • Liquefaction prediction → Maximise Recall. Missing a liquefaction-prone site (false negative) causes catastrophic failure. False positives just mean extra ground improvement cost.
  • Settlement prediction → Use RMSE in mm. R² alone is misleading — an R² of 0.9 with RMSE of 25mm may exceed your 10mm tolerance.
  • Soil classification → Use macro-averaged F1 because class imbalance is common (far more CL samples than OH samples in most databases).
  • Pile capacity → Use MAE over RMSE to avoid overweighting outlier load tests.

Water Resources: Metric Selection Guide

  • Flood early warning → Maximise Recall at threshold P = 0.3. Better to evacuate unnecessarily than miss a flood event.
  • Dam safety classification → Use weighted F1 with higher weight on "Critical" class. Use ROC-AUC for threshold-independent comparison across models.
  • Streamflow prediction → Use NSE (Nash-Sutcliffe Efficiency, equivalent to R² for hydrological models) and report RMSE in m³/s.
  • Water quality monitoring → Precision matters — false contamination alerts trigger expensive shutdowns and lab testing.
The Cost of Being Wrong

Always ask: "What happens if the model is wrong?" If a false negative means structural collapse, optimise for recall. If a false positive means wasting $50K on unnecessary testing, optimise for precision. The metric should reflect the engineering consequence.

Knowledge Check

Test your understanding of model evaluation metrics. Each question reinforces a key concept from this sub-chapter.

Quick Check 1

A crack detection model is deployed on 5,000 bridge deck images. Only 25 images contain actual cracks. The model achieves 99.5% accuracy. Which statement is most accurate?

Correct! With only 0.5% positive samples, a model that always predicts "no crack" achieves 99.5% accuracy while catching zero cracks (0% recall). Accuracy is virtually meaningless on heavily imbalanced datasets. Recall tells you whether the model actually finds the defects.
Not quite. This is the accuracy paradox. With 99.5% of images being crack-free, even a model that never detects a single crack achieves 99.5% accuracy. You must check recall to determine if cracks are being detected.

Quick Check 2

For a concrete compressive strength prediction model, you observe: MAE = 2.1 MPa, RMSE = 4.8 MPa, and R-squared = 0.91. The large gap between MAE and RMSE suggests:

Correct! When RMSE is much larger than MAE, it means a few predictions have very large errors that are amplified by the squaring in MSE. Most predictions are within ~2 MPa (the MAE), but some outliers are much worse, pulling RMSE up to 4.8 MPa. Investigating those outlier samples could reveal problematic mix designs or data quality issues.
Not quite. A large gap between MAE and RMSE indicates the presence of outlier errors. RMSE squares errors, so a few large errors inflate it disproportionately. This is useful diagnostic information, not an error.

Quick Check 3

You are building a model to screen bridges for potential seismic vulnerability. Missing a vulnerable bridge could lead to catastrophic failure in an earthquake. Which metric should you primarily optimise?

Correct! In safety-critical screening applications, recall (sensitivity) is paramount. Missing a vulnerable bridge (false negative) could lead to loss of life. False positives (flagging a safe bridge for inspection) cost money but not lives. You should maximise recall, accepting some reduction in precision as the cost of safety.
Not quite. In safety-critical applications where missing a positive case could be catastrophic, recall is the primary metric. The cost of a false negative (missed vulnerable bridge) far exceeds the cost of a false positive (unnecessary inspection).

Quick Check 4

Why is stratified k-fold cross-validation preferred over standard k-fold for a damage classification dataset where only 8% of structures are classified as "severely damaged"?

Correct! Stratified k-fold preserves the class distribution in each fold. With only 8% positive samples, random splitting could easily create folds with 0% or 15% positive samples, leading to unreliable and highly variable metric estimates. Stratification ensures each fold is representative of the overall dataset.
Not quite. Stratified k-fold preserves the class distribution in each fold. With only 8% positive samples, random splitting could produce folds with zero positive samples, making recall impossible to calculate for those folds. Stratification prevents this.

Key Takeaways

  • Accuracy alone is misleading for imbalanced datasets common in civil engineering. Always examine the confusion matrix and class-specific metrics.
  • Precision measures trustworthiness of positive predictions; recall measures completeness of detection. For safety-critical applications, recall is paramount.
  • The ROC curve and AUC provide threshold-independent model comparison. AUC represents the probability that the model ranks a positive higher than a negative.
  • For regression, use MAE for interpretability, RMSE when large errors are especially costly, and R-squared (adjusted) to understand explained variance.
  • K-fold cross-validation provides reliable performance estimates by rotating the test set. Use stratified k-fold for imbalanced classification.
  • Always translate metrics back into engineering units and validate against design code tolerances. An RMSE of 1.9 mm is meaningful; an RMSE of 0.042 is not.
