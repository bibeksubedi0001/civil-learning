Fundamentals

What is Supervised Learning? 📝

Supervised Learning is like an engineering student studying past exam papers with answer keys. The student sees the question (input) and the correct answer (output) for hundreds of examples. By studying enough examples, the student learns the underlying patterns — so when a brand new question appears on the actual exam, they can predict the answer.

The "supervised" part means a teacher provides the correct answers during training. In ML terms, this means we have labeled data — each data point has both the input features AND the known target outcome.

🌍

The Site Response Analysis Analogy

In geotechnical earthquake engineering, Site Response Analysis predicts how a specific site amplifies seismic ground motion. You know the soil profile (layers, Vs30, density, damping) and you know the recorded surface motion for past earthquakes. The ML model learns this input-output mapping, then predicts amplification for new, uncharted sites where you only have the soil profile but no earthquake recordings.

The Three Phases of Supervised Learning

1

📊 Data Collection — The Borehole Logs

Gather historical labeled data: hundreds of sites where you know both the soil conditions (input features: soil type, layer thickness, shear wave velocity, water table depth) AND the measured seismic amplification factor (target output). This is your training dataset.

2

⚙️ Training — Learning the Patterns

Feed the labeled data into an algorithm. The algorithm adjusts its internal parameters to minimize the difference between its predictions and the actual known answers. Like a student practicing — each wrong answer teaches them something, until their predictions closely match reality.

3

🎯 Prediction — The New Site

Deploy the trained model on new, unseen data. A new borehole log arrives from a project site. You feed it to the model, and it predicts the expected seismic amplification — without needing to wait for an actual earthquake or run a computationally expensive numerical simulation.

Process

The Training Process — How Models Learn ⚙️

Training a supervised model is an iterative process of predict → compare → adjust → repeat. It's exactly like calibrating a surveying instrument: you aim at a known benchmark, check the error, adjust the instrument, and repeat until the readings match the known elevation.

The Calibration Analogy

Loss function = the error between your instrument's reading and the known benchmark. Gradient descent = the direction and amount you turn the adjustment knob. Epoch = one complete round of checking all benchmarks. Learning rate = how aggressively you turn the knob (too much → overshoot, too little → takes forever).

Task Type 1

Regression — Predicting Continuous Values 📈

Regression is the task of predicting a continuous number. Not a category, not a label — a specific numerical value on a continuous scale. "What will the concrete compressive strength be at 28 days?" "What's the expected settlement of this foundation?" "What will the traffic volume be at 5 PM?"

📐

Task Type 2

Classification — Predicting Categories 🏷️

Classification predicts a discrete category or label. Not a number — a class. "Is this concrete sample acceptable or defective?" "Is this slope stable, marginally stable, or unstable?" "Is this traffic pattern normal or anomalous?"

🏗️

CE Analogy: Soil Classification

Given soil properties (grain size distribution, liquid limit, plastic limit, moisture content), classify the soil type: Gravel (GW, GP, GM, GC), Sand (SW, SP, SM, SC), Silt (ML, MH), or Clay (CL, CH). The output is a discrete category, not a continuous number.

Binary vs. Multi-Class Classification

Binary

Two Classes Only

Pass / Fail. Safe / Unsafe. Crack / No Crack. The simplest form. The model outputs a probability (e.g., 87% chance of "safe") and you set a threshold (typically 50%) to make the final decision.

Multi-Class

Three or More Classes

Soil type classification (6+ classes). Structural damage grade (None, Minor, Moderate, Severe, Collapse). The model outputs a probability for each class and picks the highest. More complex, but same principle.

Correct! The output is a specific number on a continuous scale (23.4 mm), making this regression. If it said "Safe / Unsafe," that would be classification.
Not quite. The key is the output type: 23.4 mm is a continuous number, which makes this a regression task.
Deep Dive

Types of Regression 📊

Just as there are many methods for soil stabilization (cement, lime, geosynthetics), there are many regression algorithms. Each has strengths and weaknesses depending on the data's complexity.

Simplest

Linear Regression

Fits a straight line: y = mx + b. Works when the relationship between input and output is approximately linear. Like estimating concrete strength as a linear function of water-cement ratio. Fast, interpretable, but can't capture curves.

Flexible

Polynomial Regression

Fits a curve: y = ax² + bx + c (or higher degree). Captures nonlinear relationships. Like modeling how load-displacement curves follow a parabolic path up to yield, then level off. More flexible but risks overfitting.

Regularized

Ridge & Lasso Regression

Linear regression with a penalty that prevents extreme coefficients. Ridge (L2) shrinks coefficients toward zero. Lasso (L1) can force some to exactly zero (automatic feature selection). Use when you have many correlated features.

Ensemble

Random Forest Regression

Builds hundreds of decision trees and averages their predictions. Handles nonlinear relationships, mixed feature types, and missing data. Like getting 100 independent engineering estimates and averaging them — the crowd wisdom is usually more accurate than any individual.

Deep Dive

Types of Classification Algorithms 🏷️

Each classification algorithm draws the decision boundary differently. Some draw straight lines, others draw curves, and some create complex regions. The best choice depends on your data.

Linear

Logistic Regression

Despite the name, this is a classification algorithm. It draws a straight decision boundary and outputs probabilities. Like drawing a line on a soil plasticity chart separating clay from silt. Simple, fast, interpretable, works great when classes are linearly separable.

Margin-Based

Support Vector Machine (SVM)

Finds the boundary that maximizes the margin (distance) between classes. Like finding the widest possible buffer zone between two landslide-prone areas. Can use "kernel tricks" to handle non-linear boundaries. Very effective for high-dimensional data.

Rule-Based

Decision Tree

Creates a flowchart of yes/no questions. "Is Vs30 > 300? → Yes → Is depth to bedrock < 5m? → No → Classify as Site Class D." Extremely interpretable — you can literally read the logic. But a single tree is prone to overfitting.

Ensemble

Random Forest

Builds hundreds of decision trees, each trained on a random subset of data. The final prediction is a majority vote. Like consulting 100 engineers independently — if 87 say "stable" and 13 say "unstable," you go with "stable." Robust, accurate, handles messy real-world data.

Distance-Based

K-Nearest Neighbors (KNN)

Classifies a point by looking at its K nearest neighbors and taking a majority vote. Like classifying a soil sample by finding the 5 most similar samples in your database and using their classification. Simple but can be slow for large datasets.

Deep Learning

Neural Network Classifier

Multi-layer networks that learn complex, non-linear decision boundaries automatically. Can handle any boundary shape. Like having a team of engineers where each one specializes in a different aspect — together they can classify anything. Powerful but needs more data.

Critical Concept

Overfitting vs. Underfitting ⚠️

The most important concept in all of machine learning. Get this wrong, and your model is worthless.

📏

The Surveyor's Dilemma

Imagine a surveyor measuring ground levels along a road alignment. The ground has a general slope with some natural undulation. Underfitting = drawing a perfectly horizontal line, ignoring the obvious slope. Good fit = drawing a smooth curve that captures the trend and major features. Overfitting = connecting every single measurement point with a jagged line — capturing every tiny measurement error as if it were real terrain.

Degree: 3

Underfitting (Too Simple)

The model is too simple to capture the real patterns. It performs poorly on both training data AND new data. Like using a single average value for all soil types — technically a "model," but useless for prediction.

Good Fit (Just Right)

The model captures real patterns without memorizing noise. It performs well on training data AND generalizes to new data. This is the goal. Like a well-calibrated site characterization model.

Overfitting (Too Complex)

The model memorizes the training data, including noise and random variations. It performs perfectly on training data but terribly on new data. Like a curve that passes through every measurement error — it looks perfect on paper but predicts nothing useful.

How to Prevent Overfitting

Technique How It Works CE Analogy
More Data Give the model more examples to learn from More borehole data = better site characterization
Cross-Validation Test on different subsets of data to ensure consistency Checking your design against multiple load combinations
Regularization Penalize overly complex models (Ridge, Lasso) Building code limits on span-to-depth ratios
Early Stopping Stop training before the model starts memorizing Stopping compaction before the soil starts loosening
Simpler Model Use fewer parameters, fewer layers Using a simpler structural analysis method when appropriate
Metrics

Evaluating Model Performance 📊

After training, how do you know if your model is good? In civil engineering, you validate designs against building codes. In ML, you validate models against evaluation metrics.

Regression Metrics

R² (Coefficient of Determination)

How much variance in the output does your model explain? R² = 1.0 means perfect prediction. R² = 0.0 means the model is no better than just predicting the average. Negative R² means the model is worse than useless.

MAE (Mean Absolute Error)

Average absolute difference between predicted and actual values. If MAE = 3.2 mm for a settlement prediction model, on average the model is off by 3.2 mm. Easy to interpret — it's in the same units as your output.

RMSE (Root Mean Square Error)

Similar to MAE but penalizes large errors more heavily. Like a factor of safety — small errors are tolerable, but one large error could be catastrophic. Use RMSE when large errors are unacceptable.

Classification Metrics

Confusion Matrix Explorer Click cells to adjust values

Accuracy

Percentage of correct predictions overall. If 95 out of 100 soil samples are correctly classified, accuracy = 95%. Warning: misleading when classes are imbalanced (99% of bridges are safe → always predicting "safe" gives 99% accuracy but catches no defects).

Precision

Of all the times the model said "defective," how many actually were? High precision = few false alarms. Like a crack detection system that rarely flags healthy concrete as damaged.

Recall (Sensitivity)

Of all actually defective items, how many did the model catch? High recall = few missed defects. For safety-critical applications (bridge inspection), you want very high recall — missing a real crack is far worse than a false alarm.

Knowledge Check #2

A bridge crack detection system correctly identifies 45 out of 50 actual cracks, but also flags 20 healthy spots as cracks. Which metric would be high?
A Precision — because it correctly identifies most cracks
B Recall — 45/50 = 90% of real cracks are caught
C Accuracy — overall correct predictions are high
Correct! Recall = 45/50 = 90% — it catches most cracks. But Precision would be lower: 45/(45+20) = 69% — many of its "crack" predictions are false alarms.
Think carefully: Recall asks "of all actual cracks (50), how many did we find (45)?" That's 90%. Precision asks "of all predictions of 'crack' (45+20=65), how many were right (45)?" That's only 69%.

Supervised Learning in Code 👀

Regression Example: Concrete Strength Prediction

concrete_strength.py — Supervised Regression 
# Supervised Learning: Predict concrete compressive strength
# Input: mix design parameters → Output: 28-day strength (MPa)

from sklearn.model_selection import train_test_split
from sklearn.ensemble import RandomForestRegressor
from sklearn.metrics import mean_absolute_error, r2_score

# Step 1: Load labeled data (200 concrete samples)
data = load_data("concrete_tests.csv")
# Columns: cement, water, aggregate, admixture, curing_temp → strength_28d

# Step 2: Split into training (80%) and testing (20%)
X = data[["cement", "water", "aggregate", "admixture", "curing_temp"]]
y = data["strength_28d"]
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)

# Step 3: Train the model (learning phase)
model = RandomForestRegressor(n_estimators=100)
model.fit(X_train, y_train)    # ← LEARNING HAPPENS HERE

# Step 4: Evaluate on test data (never seen during training)
predictions = model.predict(X_test)
print(f"MAE: {mean_absolute_error(y_test, predictions):.1f} MPa")
print(f"R²: {r2_score(y_test, predictions):.3f}")
# → MAE: 2.8 MPa | R²: 0.924 (excellent!)

# Step 5: Predict for a NEW mix design
new_mix = [[350, 175, 1100, 5.0, 25]]
predicted_strength = model.predict(new_mix)
# → 38.7 MPa (predicted 28-day compressive strength)
The pattern is always: Load Data → Split → Train → Evaluate → Predict. The train_test_split is critical — it ensures you test on data the model hasn't seen, preventing over-optimistic results.

Classification Example: Slope Stability

slope_stability.py — Supervised Classification 
# Classify slope stability: Stable / Marginally Stable / Unstable

from sklearn.ensemble import GradientBoostingClassifier
from sklearn.metrics import classification_report

# Labeled data: 500 slopes with known stability outcomes
features = ["slope_angle", "soil_cohesion", "friction_angle",
            "water_table", "slope_height", "rainfall_intensity"]

X_train, X_test, y_train, y_test = train_test_split(
    data[features], data["stability_class"], test_size=0.2
)

model = GradientBoostingClassifier(n_estimators=200, max_depth=5)
model.fit(X_train, y_train)

# Evaluate
print(classification_report(y_test, model.predict(X_test)))
#              precision  recall  f1-score
# Stable          0.94     0.97     0.95
# Marginal        0.82     0.78     0.80
# Unstable        0.91     0.89     0.90

# Predict for a new slope
new_slope = [[35, 15, 28, 2.5, 12, 50]]  # 35° angle, c'=15 kPa, φ'=28°...
print(model.predict(new_slope))  # → "Marginally Stable"
print(model.predict_proba(new_slope))
# → [0.12, 0.71, 0.17]  (12% Stable, 71% Marginal, 17% Unstable)
Notice predict_proba() — it gives you confidence levels, not just a label. The model says this slope is 71% likely "Marginally Stable." An engineer can use these probabilities for risk assessment, not just a binary decision.
Assessment

Chapter Summary & Final Quiz 🎯

Supervised = Labeled Data

The model learns from input-output pairs where the correct answer is known. It maps patterns from inputs to outputs.

Two Task Types

Regression (predict numbers) vs Classification (predict categories). Same learning process, different output types.

Overfitting is the Enemy

A model that memorizes training data is useless. Use validation, regularization, and proper train-test splitting.

Evaluation Matters

R², MAE for regression. Accuracy, Precision, Recall for classification. Always evaluate on unseen test data.

Final Challenge

You train a crack detection model that achieves 99% accuracy on the training data but only 62% accuracy on new bridge photos. What is the most likely problem?
A Underfitting — the model is too simple
B Overfitting — the model memorized the training data instead of learning general patterns
C The training data was too large
D The model needs more epochs of training
Exactly! 99% on training but 62% on new data is the textbook definition of overfitting. The model memorized the specific training photos instead of learning what cracks actually look like. Solutions: more diverse training data, regularization, simpler model, or data augmentation.
The key clue: high training accuracy + low test accuracy = overfitting. The model performs perfectly on data it has seen (memorization) but poorly on new data (no generalization). This is the opposite of underfitting.
